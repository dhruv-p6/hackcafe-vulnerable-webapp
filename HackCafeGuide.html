// vulnerability 01 of 07 — search bar
Reflected XSS
(Cross-Site Scripting)

The search bar takes what you type and puts it directly onto the page as HTML. That means you can type HTML tags — and the browser will run them.

// what is XSS?

XSS stands for Cross-Site Scripting. It happens when a website takes text you typed in and puts it on the page without checking if it contains HTML or JavaScript.

Imagine a name tag that says "Hello, [your name]". If the website just pastes your input in directly, and you type <b>Bob</b> — the browser will make it bold instead of showing the angle brackets. Now imagine typing JavaScript instead.

"Reflected" means the attack bounces off the server back to you immediately — like a search result. It's different from "Stored" XSS (Vuln #6) where the attack is saved and shown to other visitors too.

// why does this vulnerability exist?
// the vulnerable code (in hackcafe.html)
// ❌ VULNERABLE: user input pasted directly into innerHTML
function doSearch() {
  const q = document.getElementById('searchInput').value;
  document.getElementById('searchResults').innerHTML =
    `Results for: <strong>${q}</strong>`; // ← q goes in raw
}
📖
The problem is .innerHTML. It tells the browser "treat this as HTML". Using .textContent instead tells the browser "treat this as plain text" — which makes it safe. Simple fix, big difference.
The Fix
// safe version
// ✅ SAFE: use .textContent — browser treats it as plain text
function doSearch() {
  const q = document.getElementById('searchInput').value;
  document.getElementById('searchResults').textContent =
    `Results for: ${q}`; // angle brackets shown as text, not run
}
✅
Remember: Any time you put user input on a webpage, use .textContent (or escape HTML entities). Never use .innerHTML with untrusted input.
// vulnerability 02 of 07 — login form
SQL Injection
(SQLi)

The login form takes your username and password and puts them directly into a database query. By adding a special character, you can change the meaning of that query — and skip the password check entirely.

// what is SQL injection?

Websites that have accounts store your username and password in a database. When you log in, the site runs a database query like: "find me the user where name is 'alice' AND password is 'abc123'".

SQL Injection means you type something that changes the query itself. The magic character is a single quote '. It breaks out of the text part of the query and lets you add your own SQL commands — like telling the database to ignore the password check.

It's one of the oldest and most dangerous web vulnerabilities. It can expose entire databases, not just one account.

"SQL injection for beginners"
"what does -- mean in SQL"explains the comment trick

// step-by-step exploit
1
Open the login form on HackCafe
Click the Login button in the top-right of HackCafe. A modal popup will appear.
2
Type the injection payload into the Username field
Leave the password field empty (or type anything). Put this in the Username field only:
// the SQL injection payload — username field only
admin' --

That's: the word admin, then a single quote ', then two dashes --, then a space. The space at the end matters.

3
Click Sign In — you're in!
You should see "Logged in as admin (admin) — via SQL injection bypass!" — you got into the admin account without knowing the password. Screenshot this.
// why does admin' -- work?
// how the backend query gets manipulated
-- What the server SHOULD run:
SELECT * FROM users
WHERE username = 'alice' AND password = 'abc123'

-- What it runs with our payload (admin' -- ):
SELECT * FROM users
WHERE username = 'admin' --' AND password = ''
-- ↑↑ everything after -- is a COMMENT
-- so the password check is IGNORED
💡
In SQL, -- starts a comment — everything after it is ignored. So our payload closes the username string early with ', then comments out the rest of the query. The database just finds the admin user and returns them, no password needed.
The Fix
// parameterized query — never concatenate user input into SQL
// ✅ SAFE — values passed separately, never part of the SQL string
cursor.execute(
  "SELECT * FROM users WHERE username = %s AND password = %s",
  (username, password) ← passed as data, not code
)
// vulnerability 03 of 07 — order lookup
IDOR
(Insecure Direct Object Reference)

The order lookup page takes a number you type in and shows you that order — without checking if it belongs to you. Just change the number to see someone else's order.

// what is IDOR?

IDOR stands for Insecure Direct Object Reference. It's a fancy name for a simple problem: the website uses a number (like an order ID) to find data, but doesn't check whether you're allowed to see that data.

Think of it like a coat check — you hand in your ticket (number 47) and get your coat. IDOR means if you hand in number 48, you get someone else's coat. The system just grabs whatever number you give it.

This is one of the most common vulnerabilities in real apps. It's responsible for a lot of data breaches where user data gets exposed.

// step-by-step exploit
1
Find the Order Lookup section on HackCafe
Scroll down past the menu items on HackCafe. You'll see a section called Order Lookup with a number input box. The default is 3 (which is your order).
2
First, look up your own order (ID 3)
Click Look Up with the number 3. You see Sarah's order — that's "your" order in this scenario. Name, email, item, total, status.
3
Now change the number to 1, 2, 4, or 5
Change the number to 1 and click Look Up again. You'll see Alice's order — her name, her email, what she ordered. You're reading someone else's private data just by changing a number. Try 2, 4, and 5 too.
⚠️
Order ID 4 is the admin's internal order. It reveals their email address. In a real app, this kind of enumeration could expose every user's purchase history, home address, and payment info.

// vulnerable code — no ownership check

  const order = orders.find(o => o.id === id);

// vulnerability 04 of 07 — html source code
Sensitive Data
Exposure

Developers sometimes leave passwords, API keys, or notes in the HTML source code as comments. These are invisible on the page but visible to anyone who views the source.

// what is this vulnerability?

Every webpage is made of HTML code that your browser downloads and then displays. Normally you just see the pretty version — but you can also view the raw HTML source by pressing Ctrl+U.

In HTML, a comment is text that starts with <!-- and ends with -->. It's invisible on the page but completely readable in the source. Developers use them to leave notes to themselves — and sometimes forget to remove sensitive information before publishing.

This is simpler than the other vulnerabilities here: you don't type anything special. You just read the source code.

// step-by-step exploit
1
View the page source on HackCafe
On the HackCafe page, press Ctrl+U (Windows/Linux) or Cmd+U (Mac). A new tab opens showing the raw HTML of the page. Alternatively, right-click anywhere and choose "View Page Source".
2
Search for HTML comments
In the source tab, press Ctrl+F and search for <!--. This finds all comments. Click through each result.
// what you'll find

Near the top of the source and in the footer, you'll find comments like these:

<!-- TODO: remove creds before going live -->
<!-- admin login: admin / letmein -->
<!-- stripe test key: sk_test_ABCDEF123456 -->

Now go to the Login modal and try admin / letmein — it works! The comment just gave you the password.

3
Also check the JavaScript
Press F12 → Sources tab → open the HTML file. You can see the entire JavaScript code. A real attacker would read through this to understand how the app works and find more vulnerabilities.
⚠️
This also happens in git history. If you committed a password to GitHub and then deleted it, it's still visible in the commit history. Use git log to check, or tools like truffleHog. Search: "how to remove sensitive data from git history"
The Fix
✅
Never put real credentials, keys, or connection strings in HTML comments or source code. Store secrets in environment variables (a .env file), and add .env to your .gitignore so it never gets committed. Search: "how to use .env files in node.js"
// vulnerability 05 of 07 — login form
Weak Passwords &
Default Credentials

The app uses simple, guessable passwords. You don't need any special tricks — just try common passwords and you're in.

// what is this vulnerability?

Weak credentials means the app's passwords are easy to guess. Either the developer hardcoded simple passwords for test accounts and forgot to change them, or users chose terrible passwords and the app let them.

This is one of the most common reasons real accounts get compromised. It doesn't require any fancy hacking — you just try a list of popular passwords until one works. This is called a brute force or credential stuffing attack.

The passwords in HackCafe are also stored in plain text — meaning if you get into the database, you can read every user's real password directly.

// step-by-step exploit
1
Open the Login modal
Click Login in the HackCafe navigation. The modal appears.
2
Try these common/weak passwords
Try each of these username/password combinations. They all work:
// all working credentials (all weak/guessable)
admin / letmein
alice / password
bob / 123456
sarah / qwerty

"letmein", "password", "123456", "qwerty" — these are all in the top 10 most common passwords in the world.

3
Open the Admin Panel to see passwords stored in plain text
Click the Admin Panel button. You can see every user's password in plain text in the table — not even hashed. If this were a real database breach, an attacker would have everyone's password immediately.
💡
In real apps, passwords should be stored as hashes — a scrambled version that can't be reversed. The app checks your login by hashing what you type and comparing it to the stored hash. Use bcrypt — it's slow enough to resist brute-force attacks. Search: "bcrypt password hashing explained"
The Fix
✅
For users: Enforce minimum 12 characters, mixed case, numbers. Block known common passwords (use the haveibeenpwned API).
For developers: Store passwords using bcrypt with a cost factor of 12+. Never store plain text. Add rate limiting to block brute force (max 5 attempts before lockout). Search: "how to use bcrypt in node.js"
// vulnerability 06 of 07 — comment / review section
Stored XSS

Like Reflected XSS (Vuln #1), but worse. Your injected code gets saved and runs for every visitor who loads the page — not just you.

// what makes stored XSS different?

In Reflected XSS (Vuln #1), the injected script only affects the person who submits it — the attack bounces back at you only. In Stored XSS, your script gets saved to the database and shown to everyone who visits the page afterward.

Imagine a comment on a blog post that contains a script that steals cookies. Every visitor to that blog post would have their cookies stolen — including the admin. This is how a lot of real account takeovers happen.

This vulnerability is in the Customer Reviews section on HackCafe, where both the name and comment fields are rendered as raw HTML.

⚠️

The Fix
// escape HTML before rendering saved content
// ✅ Escape HTML entities when rendering user-generated content
function escapeHtml(str) {
  return str
    .replace(/&/g, '&amp;')
    .replace(/</g, '&lt;')
    .replace(/>/g, '&gt;');
}

// Then use it when rendering comments:
el.innerHTML = escapeHtml(comment.text); // or just use .textContent
// vulnerability 07 of 07 — admin panel
Unprotected Admin Panel

The admin panel is linked in the navigation bar and has no login check. Anyone who clicks it gets full access to all user data — no password needed.

// what is this vulnerability?

An admin panel is supposed to be the most protected part of a website — it shows all user data and lets administrators manage the site. A security misconfiguration means this protection was either never set up or was set up incorrectly.

In this case, there is literally no authentication check at all. Any visitor who clicks the "Admin Panel" button gets full access. In real apps, attackers look for admin panels by guessing common paths like /admin, /admin/login, or /dashboard.

This is probably the most beginner-friendly vulnerability in this lab — you exploit it by just clicking a button.

// why does this happen in real apps?
// vulnerable vs safe admin route
// ❌ VULNERABLE — route has no middleware
app.get('/admin', adminController);

// ✅ SAFE — requireAdmin middleware checks session first
app.get('/admin', requireAdmin, adminController);

function requireAdmin(req, res, next) {
  if (!req.session?.user) return res.redirect('/login');
  if (req.session.user.role !== 'admin') return res.status(403).send('Forbidden');
  next(); // only admins get here
}
